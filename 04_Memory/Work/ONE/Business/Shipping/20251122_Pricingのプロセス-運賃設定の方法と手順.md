# Pricingのプロセス-運賃設定の方法と手順

## 概要
ONE社におけるPricing（運賃設定）のプロセスについて説明します。Eagle-X Pricing toolを使用した運賃計算から、顧客への提示まで、Pricingの全体像を把握できます。

## 詳細

### Pricingとは
- **Pricing = 運賃設定**
- 顧客に対して提示する運賃を決定するプロセス
- 契約（Award）やスポット案件に対して適切な運賃を設定
- 採算性（CM/TEU）を確保しながら、市場競争力のある運賃を提示

### Pricingの目的
1. **採算性の確保**: CM/TEU = 660 USD（または66 USD/t）を目安
2. **市場競争力**: 競合他社と比較して適切な運賃を設定
3. **顧客満足**: 顧客の要望に応じた運賃構造を提供
4. **収益最大化**: 船腹の最適配分と収益の最大化

## Pricingの全体プロセス

### ステップ1: データの準備と取得

#### データソース
- **PRISM**: Pricing Simulation（運賃シミュレーション）
- **COA**: Lifting等の実績（過去の船積み実績）
- **DMT**: Tariff（運賃表）
- **TenderDB**: Contractの情報（契約データ）
- **Salesforce**: FAK・MRGレートの管理

#### 顧客情報の取得
- **BCO（Beneficial Cargo Owner）**: Contract Noで検索
- **NVO（Non-Vessel Operating）**: NVとNACの組み合わせで検索
- **過去のLifting実績**: 過去の取引実績を確認
- **現在のContract**: 既存契約の内容を確認

### ステップ2: Eagle-X Pricing Toolの使用

#### Pricing Toolの起動
1. **共通フォーマットをインポート**
   - Tradeごとにテンプレートが用意されている
   - MRGやサーチャージがTradeごとに異なる

2. **Search Data**
   - 顧客を選択（NVOを選ぶと横にNACが出てくる）
   - Tradeを選択
   - Link finalで30_2のリンクもシートに持つ（サーチ時間短縮のため）

3. **Get DataでRetrieve**
   - **Lifting**: 過去の実績を取得
   - **Route**: 該当Port pairのRoute候補をすべて取得
   - **コスト**: 各タイプ・サイズごとのコストも取得

#### Pricing Toolの基本操作フロー

```
1. Set END to END（最初に押下）
   ↓
   ミニマムコストのルートをセット
   End to End costが出てくる

2. T&Cを読む
   ↓
   Rate Structureの解釈
   顧客の契約条件を理解

3. Set Surchargeを押下
   ↓
   All rowsにPopulate
   サーチャージを設定

4. Update Conditionボタンを押下
   ↓
   右側のサーチャージカラムが反映される
   （InclusiveがSubject toか）

5. Retrieve MRGを押す
   ↓
   MRGカラムが埋められる
   Long Term MRGのRate Structureも埋められる

6. FRT Control Panel
   ↓
   Benchmarkを選ぶ（MRG、Target CM、Current Contract）
   Surcharge Updateを押す

7. All in Calculationを押す
   ↓
   すべてのサーチャージをたされた総額がでる

8. Base Rateの計算
   ↓
   All inカラムーSubjectサーチャージが計算される
```

### ステップ3: 運賃の計算

#### 重要な概念

| 概念 | 説明 |
|------|------|
| **LT MRG** | Long Term Minimum Rate Guideline（長期最低料金ガイドライン）<br>Trade Managementが長期顧客向けに設定した最低料金水準 |
| **Target CM** | Target Contribution Margin（目標限界利益）<br>Trade Managementが設定した、長期顧客向けの料金提示の基準となる最低限界利益水準 |
| **Current Contract** | 顧客とONE社との間で合意された特定の港間ペアにおける料金水準 |
| **Benchmark Rates** | マーケティング／価格設定部門の提示の基準となる料金<br>LT MRG、Target CM、Current Contractに基づく |
| **Customer's Request** | 顧客の契約条件の解釈に基づくマーケティング／価格設定部門が定義した料金構造 |
| **All In Calculation** | Benchmark Rates + Benchmark Rate Structure（Subject）<br>すべてのサーチャージをたした総額 |
| **Base Rate** | マーケティング／価格設定部門が顧客に提示しようとする料金水準<br>All In Ratesから顧客の要望する料金構造を差し引いて算出 |

#### 運賃計算のロジック

```
1. Benchmark Ratesを決定
   - LT MRG、Target CM、Current Contractから選択
   - Pricer、Tradeが判断

2. All In Calculation
   - Benchmark Rates + Benchmark Rate Structure（Subject）
   - Apple to Apple比較のため

3. Base Rateの算出
   - All In Rates - Customer's Request Rate Structure
   - 顧客が求めている部分のレート

4. 採算性の確認
   - Expected CM/TEUを計算
   - CM/TEU = 660 USD（または66 USD/t）を目安
```

#### FRT Control Panelでの操作

1. **Benchmarkを選ぶ**
   - MRG、Target CM、Current Contractから選択
   - 選択した値をSUMする式が入る

2. **Surcharge Updateを押す**
   - サーチャージの方もLong Term MRGのRate Structureが反映される

3. **Add on insert**
   - お客さんごとに追加が必要だったら追加
   - 例えばDMT等が追加できる

4. **Decline offer**
   - MRGがうまっていないところはDecline offerにYをいれる
   - Decline reasonが入れられる

5. **Base Rateの選択**
   - FRTコントロールパネルの右側で選択
   - 差し引きたいSurchargeを選択できる
   - Banker関連のSurchargeが引かれてBase Rateが出てくる

### ステップ4: 料金体系の理解

#### 料金の種類

| 料金タイプ | 説明 | 適用対象 |
|----------|------|---------|
| **FAK** | Freight All Kinds（すべての一般貨物に適用する基準運賃） | すべての一般貨物 |
| **Tier-1** | FAKから▲100 USD/TEU | 安定して週30 TEU×3か月以上出荷する顧客 |
| **Short Term** | 〜3か月（市況連動・スポット対応） | 短期契約 |
| **Mid Term** | 3–6か月（四半期ごとの量確保） | 中期契約 |
| **Long Term** | 6か月超（BCO年間入札・収益安定） | 長期契約 |

#### All inとSuper All inの違い

- **All in**: Base Freightに主要な海上サーチャージ（BAF、OFC、LSSなど）をまとめて含めた料金
- **Super All in**: All inよりさらに広く、ほとんどのサーチャージを含めた完全一括運賃

#### サーチャージの扱い

- **サーチャージは23個裏では持っている**
- **フィールドマッピングで指定された5つだけを持っている**
- **OBSはSYMが管理**
- **ローカルサーチャージは各国のSYMが管理**
- **四半期に一回洗い直す**
- **デフォルトでInclusiveとなっている**（Tigerと同じ）
- **Subject toにしたいのだけ選ぶ**

### ステップ5: 採算性の確認

#### CM/TEUの計算

- **Expected CM/TEUはコストを元に計算されている**
- **MRGがないレーンやIntra AsiaやNondomiで使う**
- **重要なコンポーネントだけを指定するFlexibilityを持たせている**

#### 採算ボーダー

- **CM/TEU = 660 USD（または66 USD/t）**
- この基準を下回る場合は、特別な理由が必要

### ステップ6: 顧客への提示

#### Export to Customer Sheet

1. **Routeの確認**
   - Routeの場所を変えた場合は黄色でハイライトされている

2. **Base Rateの確認**
   - Base Rateのクリームの色の部分が顧客が求めている部分のレート
   - Offerしたくない場合はNo Offerを選択する

3. **必須項目の確認**
   - Y = Optional
   - Y/M = Mandatory

4. **Submitボタン**
   - このシートだけ出力
   - File infoに出力先等のデータを持っている
   - 今後メールもできる（Gフォーム経由）

#### 重要なポイント

- **顧客がAll Inしか求めていないときも常に最後のBase Rateまでオペレーションすることが求められる**
- **顧客のレートストラクチャーの前提で全部足しこんだものと、LTMRG前提で全部足したものを比較して、最終的にBase Rateを算出している**
- **Base Rateが最終的にCustomer Sheetに反映される**
- **All in rateはApple to Apple比較のため**

## 特殊なケース

### Base PortとOut Port

- **Base Port**: 主要航路上の拠点となる港（例: シンガポール、釜山、横浜、ロッテルダム、ロサンゼルス）
- **Out Port**: Base Port以外の港で、追加コスト（追加運賃）が発生する可能性がある港
- **Outport Surcharge**: Out Portの場合、追加費用が発生

### EUA IOPとIndia Inland haulage

- **EUA IOP**: EuropeとAfricaはSeparate IOP toolがある
- **India Inland haulage**: インドの内陸のコスト（主にRail）はインドMarketingが決めている
- **PODを変更するたびにデータが変わる**
- **全Tradeで使用可能**

### SOC（Shipper's Own Container）

- **自社コンテナ優先のため受入は限定的**
- **FAK＋100 USD/TEUが目安**

## よくある質問

### Q: Pricing Toolを使わずにPricingできますか？
**A**: 可能ですが、推奨されません。Pricing Toolを使用することで、正確な運賃計算と採算性の確認が可能になります。

### Q: MRGがない場合はどうしますか？
**A**: Expected CM/TEUを使用します。MRGがないレーンやIntra AsiaやNondomiで使います。

### Q: Subject to allのSurchargeはどこから持ってくるか？
**A**: Applicable Surcharge from PRISMから取得します。Pricerが高いレートを出したいならこっちを使ってもいいですが、Customer's requestはPricerがCustomerのリクエストを再現するべきです。

### Q: Base RateとAll In Rateの違いは？
**A**: 
- **All In Rate**: すべてのサーチャージをたした総額（Apple to Apple比較のため）
- **Base Rate**: 顧客に提示する料金水準（All In Ratesから顧客の要望する料金構造を差し引いたもの）

### Q: Pricingの承認プロセスは？
**A**: Tigerを使用して、前線営業からのレートリクエストを承認・却下します。承認後、自動でRFA Proposalに登録されます。

## ベストプラクティス

1. **データの正確性**: 常に最新のデータを使用する
2. **採算性の確認**: CM/TEU = 660 USD（または66 USD/t）を目安にする
3. **顧客の要望を理解**: T&Cを正確に読み、Rate Structureを解釈する
4. **比較検討**: All In RateでApple to Apple比較を行う
5. **Base Rateまで計算**: 顧客がAll Inしか求めていないときも常に最後のBase Rateまでオペレーションする
6. **ドキュメント化**: Pricingの根拠を記録する
