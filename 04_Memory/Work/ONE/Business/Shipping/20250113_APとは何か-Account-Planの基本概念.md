# APとは何か-Account Planの基本概念

## 概要
AP（Account Plan）は、ONE社における顧客別の船腹配分計画を管理するシステムです。各顧客に対して、どのサービス（航路）にどれだけのコンテナ（TEU）を割り当てるかを月次で計画し、実際の船積み実績と比較・管理します。

## 詳細

### APの定義
- **AP = Account Plan（アカウントプラン）**
- 顧客別の船腹配分計画
- 月次で作成・更新される（TP tradeは隔週）
- 各顧客に対して、どのサービスにどれだけのTEUを割り当てるかを決定

### APの目的
1. **船腹の最適配分**: 限られた船腹を顧客間で適切に配分
2. **需要予測**: 各顧客の需要を予測し、船腹計画に反映
3. **実績管理**: 計画と実績を比較し、差異を分析
4. **営業支援**: Sales担当者が顧客との交渉に使用するデータを提供

### APの対象範囲
- **Trade**: LAWC（Latin America West Coast）、TP（Trans-Pacific）、AE（Asia-Europe）など
- **Service**: AX1、AX2、AX3、AX4などのサービス
- **顧客**: BCO（Beneficial Cargo Owner）、NVO（Non-Vessel Operating）など

### APの主要コンポーネント
1. **BSA（船腹保証）**: 各顧客に保証する船腹量
2. **Sales Target**: 営業目標（Cimonから取得）
3. **Actual Lifting**: 実際の船積み実績
4. **Forecast**: 需要予測

## 基本的な使い方

### APの作成タイミング
- **通常**: 月次（例: 5月に6月のAPを作成）
- **TP trade**: 隔週（Biweekly basis）
- **作成期間**: 前月末～当月20日頃

### APの作成プロセス
1. Sales Targetの取得（Cimonから）
2. BSAの設定
3. 各顧客への配分計画の作成
4. OPUSへのアップロード
5. 実績との比較・分析

## 高度な使い方

### APとSpace Controlの連携
- APで計画した配分をSpace Controlに反映
- 実際のBooking状況と比較
- Roll over（積み残し）の管理

### APとPricingの連携
- APで計画した配分に基づいてPricingを決定
- 顧客の需要に応じた料金設定

## メリット・デメリット

### ✅ メリット
- 船腹の最適配分が可能
- 顧客別の需要予測ができる
- 実績と計画の比較分析ができる
- Sales担当者の営業活動を支援

### ❌ デメリット
- まだたくさんのManual workが残っている
- データの更新に時間がかかる
- 複数のシステム間でのデータ連携が複雑

## 使用例

### 例1: 月次APの作成
```
1. 前月末にSales Targetを取得
2. 各顧客のBSAを設定
3. 各サービスへの配分を決定
4. OPUSにアップロード
5. Sales担当者に共有
```

### 例2: APの更新
```
1. 実績データを確認
2. 計画との差異を分析
3. 必要に応じて配分を調整
4. Master Constraint Tableを更新
```

## よくある問題と解決策

### 問題1: 実績と計画の大きな差異
- **原因**: 需要予測の誤り、顧客の急な需要変化
- **解決策**: 定期的な実績確認と計画の見直し

### 問題2: Manual workが多い
- **原因**: 複数のシステム間でのデータ連携が手動
- **解決策**: DX taskforceによる自動化ツールの導入（進行中）

### 問題3: データの不整合
- **原因**: 複数のデータソースからの取得
- **解決策**: データの一元管理とバリデーション

## ベストプラクティス

1. **定期的な更新**: 月次で必ず更新する
2. **実績との比較**: 計画と実績を常に比較する
3. **早期の警告**: 大きな差異がある場合は早期に警告
4. **データの整合性**: 複数のデータソースの整合性を確認
5. **Sales担当者との連携**: 営業活動を支援する情報を提供
